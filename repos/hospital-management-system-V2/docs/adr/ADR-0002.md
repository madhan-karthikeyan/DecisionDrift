## Context

The system stores structured data for patients, doctors, appointments, treatments, and payments. A consistent data access layer is required to prevent scattered raw SQL queries, inconsistent transaction handling, and schema drift across the codebase.

## Decision

All database access must use SQLAlchemy ORM through Flask-SQLAlchemy. Models inherit from `db.Model`, relationships use `db.relationship()`, and queries use the ORM query API or `db.session.execute()`. Raw SQL strings are not permitted outside migration files.

## Consequences

The ORM provides consistent transaction management, migration support via Alembic, and protection against SQL injection. Model definitions serve as a single source of truth for the schema. The main tradeoff is a slight abstraction overhead for complex queries, which should use the ORM's SQL expression language rather than raw strings.
