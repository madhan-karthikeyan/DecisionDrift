## Context

The system performs several operations that should not block HTTP responses: sending emails, generating reports, exporting data, and sending reminders. Synchronous execution would degrade API response times and create a poor user experience.

## Decision

Use Celery with Redis as both broker and result backend for asynchronous task processing. Long-running or non-critical operations are dispatched via `task.delay()` and executed by a separate Celery worker process. Tasks are organized by domain in the `backend/tasks/` package.

## Consequences

API response times improve because blocking operations are offloaded to workers. Celery provides retry policies, task scheduling via Celery Beat, and fan-out patterns via `group()`. The tradeoffs are increased operational complexity (a running worker process and Redis dependency) and the need to handle task failures gracefully with retries and logging.
