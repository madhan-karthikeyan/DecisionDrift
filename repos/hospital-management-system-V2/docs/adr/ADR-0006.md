## Context

Frequently-read data like department lists, doctor directories, and available slots are queried repeatedly. Without caching, the same database queries execute on every request, adding unnecessary load.

## Decision

Use Redis as a read-through cache layer. Frequently accessed, infrequently changed data is cached with a TTL using `cache_get()` / `cache_set()`. When data is mutated, cache invalidation is explicitly triggered via functions like `invalidate_on_doctor_change()`. The cache degrades gracefully: if Redis is unavailable, the system falls back to direct database queries.

## Consequences

Database load is reduced for read-heavy endpoints. The explicit invalidation pattern prevents stale data from persisting beyond mutations. The main risk is cache staleness if invalidation is missed during a code change, which is mitigated by short TTLs on most cached data.
