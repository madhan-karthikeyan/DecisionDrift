## Context

The API serves multiple domains (authentication, admin management, doctor workflows, patient self-service, public data). As the codebase grows, a flat route structure becomes difficult to navigate and maintain.

## Decision

Organize API routes using a hierarchical Blueprint architecture. A master `api_bp` at `/api` mounts five domain sub-blueprints: `auth`, `admin`, `doctor`, `patient`, and `public`. Each sub-blueprint lives in its own package with a `routes.py` file, and its `__init__.py` re-exports the blueprint for clean registration.

## Consequences

Domain separation makes the route structure navigable and each domain independently testable. Adding a new domain means creating a new blueprint package without modifying existing routes. Developers must be disciplined about keeping logic in the correct domain blueprint rather than creating cross-blueprint dependencies.
