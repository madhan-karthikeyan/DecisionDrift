## Context

The system serves three distinct user roles with different permission levels: administrators (full access), doctors (appointment and diagnosis management), and patients (self-service). Endpoints must enforce that users can only access resources appropriate to their role.

## Decision

Enforce role-based access control at the view layer using custom decorators. The `User` model stores a `role` field, and the JWT token embeds the role at login. Decorators (`require_admin`, `require_doctor`, `require_patient`) check the role before allowing access to the endpoint, returning 403 Forbidden if the role does not match.

## Consequences

Role checks are explicit and visible at the point of use rather than buried in request handlers. The decorator pattern makes it easy to add new role-gated endpoints consistently. The main limitation is that role checks are coarse-grained — if fine-grained object-level permissions are needed later, a more elaborate permission system would need to be introduced.
